# Design Constraints: AWS Secrets Manager + External Secrets Operator

## Approach

Certificates generated by cert-manager are stored in AWS Secrets Manager as the central source of truth. The External Secrets Operator (ESO) synchronizes these certificates into Kubernetes TLS secrets across multiple namespaces.

## Architecture

```
cert-manager → K8s Secret → Push to AWS SM → ESO polls → K8s Secrets (per namespace)
```

## Dependencies

| Component | Version | Purpose |
|-----------|---------|---------|
| External Secrets Operator | >= 0.9 | Syncs AWS SM secrets to K8s |
| AWS Secrets Manager | N/A | Central secret storage |
| IAM Role (IRSA) | N/A | ESO authentication to AWS |
| cert-manager | >= 1.13 | Certificate generation |

## IAM Permissions Required

```json
{
  "Effect": "Allow",
  "Action": [
    "secretsmanager:GetSecretValue",
    "secretsmanager:DescribeSecret"
  ],
  "Resource": "arn:aws:secretsmanager:<region>:<account>:secret:istio-tls-*"
}
```

## Operational Considerations

### Certificate Rotation
- cert-manager renews the certificate and updates the K8s secret
- A separate process (CronJob or controller) must push the updated cert to AWS SM
- ESO polls AWS SM at the configured `refreshInterval` (default: 1h)
- **Total propagation time**: cert-manager renewal + push delay + ESO refresh interval

### Refresh Interval Trade-offs
- Shorter interval: Faster propagation, more AWS API calls (cost)
- Longer interval: Lower cost, slower propagation of rotated certs
- Recommended: 1h for production, 5m for testing

## Limitations

1. **Push mechanism required**: cert-manager doesn't natively push to AWS SM — requires custom automation
2. **AWS API costs**: Each ESO refresh per namespace = 1 GetSecretValue API call
3. **IAM complexity**: Requires IRSA setup with correct trust policies
4. **Single point of failure**: If AWS SM is unavailable, no new secrets are synced (existing ones remain)
5. **Eventual consistency**: Secrets are not immediately available after rotation

## Advantages

1. **Centralized management**: Single source of truth in AWS
2. **Audit trail**: AWS CloudTrail logs all secret access
3. **Cross-cluster**: Same secret can be shared across multiple EKS clusters
4. **Access control**: Fine-grained IAM policies per secret
